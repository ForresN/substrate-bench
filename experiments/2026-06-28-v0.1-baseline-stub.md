# Experiment 002 — v0.1 contract upgrade, baseline (offline `stub`)

- **Date:** 2026-06-28
- **Author:** Cognitive Systems Lab
- **Task set:** `v0` (46 tasks)  ·  **Conditions:** A–E  ·  **Solver:** `stub` (deterministic, offline)
- **Command:** `substrate-bench run --condition all --tasks v0`
- **Artifacts:** [`results/v0-stub.json`](../results/v0-stub.json), [`leaderboard.md`](../leaderboard.md)
- **Governing spec:** [`measurement-contract-v0.1.md`](../measurement-contract-v0.1.md)

## Hypothesis

The v0.1 upgrade should (a) preserve the substrate asymmetry, and (b) make the
**answer/substrate decoupling** visible — the thing v0's stub could barely show.
Specifically: splitting strategy from execution medium and forcing a route
declaration should let a condition be answer-correct yet substrate-wrong (and
vice-versa), and indiscriminate code (C) should look *worse* at scale because
`exact_computation` is the wrong strategy for most task types.

## What changed from v0 (contract §1–§5, §8)

- **Two-axis ontology.** `code` removed as a strategy; `social` added; orthogonal
  `executes_code` flag. The 10 v0 tasks were re-checked individually and remapped
  (e.g. `social-001` language→**social**; `rule-001` → **search**; `sim-002` →
  `[simulation, exact_computation]`; all `code` golds → `exact_computation`).
- **Route declaration.** Solvers emit `{strategy, executes_code, rationale,
  answer}`; A/B/C fixed, D/E emitted; `CallableModel` parses a real declaration
  and fails loudly on malformed output.
- **Two-level scoring + consistency audit** (`executes_code:true ⇒ code ran`;
  structural audit hook recorded for v0.2).
- **Baseline registry** (`baselines/`, `baseline show|promote`, `gate`) with the
  **human-only promotion guardrail**.
- **Task growth** to 46 (~5 per strategy × difficulty for the reference-backed
  cells, ~2–3 for the human-review language/social cells this sprint).

## Interpretation — the decoupling is now visible (H confirmed)

- **B: 33% answers, 15% substrate.** Chain-of-thought answers every linguistic
  task (15/46), but B declares `language` even on `social` tasks, so it is
  substrate-correct on only the 7 true `language` tasks. Answer-right /
  substrate-wrong, exactly the matrix cell v0 couldn't produce.
- **A: right strategy, wrong answer.** On hard (`difficulty ≥ 2`) `language` tasks
  A has the correct strategy but no CoT, yielding 5 `right_substrate_bad_execution`
  results — substrate-correct, answer-wrong.
- **C is wrong-strategy on ~74%.** Declaring `exact_computation` everywhere is the
  wrong strategy for `language`/`social`/`simulation`/`search`/`verify`; C's
  substrate accuracy collapses to 26% at scale, sharper than v0's 50%.
- **E's switch survives the remap.** `meta-001` (gold `exact_computation`, router
  trapped into `language`) is solved only by E, which verifies, refutes, and
  re-routes — switch rate 100% vs D's 0%.
- **Audit 100%** for the honest stub; the seam is exercised and `test_audit.py`
  covers the failing case (declared code, none ran).

As in v0, a large cross-condition composite win does not auto-merge — the gate is
candidate-vs-*same-baseline*, and the +15% cost clause binds. This is the policy
the contract confirms (§5), not a bug.

## Process note — the guardrail held

The recorded baseline for the gate is **not** auto-promoted in this run: the build
agent is bound by the same human-only guardrail as the autonomous loop (contract
§4), so `baselines/v0__stub.json` is left for a human to promote interactively
(`substrate-bench baseline promote results/v0-stub.json --solver stub --task-set
v0`). `tests/test_registry.py` asserts a non-interactive promote raises
`PromotionDenied`.

## Validation

- **87 tests green** (`pytest -q`), incl. the scoring split, audit pass/fail,
  registry read-only guardrail, declaration parse + loud-failure, and the
  reference-derivation provenance proof for all 33 `reference` tasks.
- **Byte-identical re-runs** of `results/` and `leaderboard.md` confirmed.

## Threats to validity / what this does *not* show

- The `stub` remains strategy-gated by construction (offline fixture, §8); the
  decoupling shown is the *designed* stub behaviour, not a model measurement.
- 46 tasks; the human-review `language`/`social` cells are at ~2–3 (not yet 5).
- The audit is medium-only; the structural audit (does a `search` declaration
  actually expand states) is deferred to v0.2.

## Next steps

   accuracy to decouple *further* and C's penalty to persist.
2. Stand up the **review-gated task-generation loop** (contract §7) with
   separation of powers: the generator is never the solver under test.
3. Implement the v0.2 structural audit at the recorded hook.
